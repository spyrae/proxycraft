# ADR 005: External Multi-Region Geo Probes

## Status

Accepted

## Context

`RB-537` and `RB-540` established deterministic smoke fixtures and deploy-blocking post-deploy verification. That closed the gap between "the service is up" and "the core code path still works from production", but it still did not answer the user-facing question that matters for VPN/proxy products:

1. Are the public endpoints reachable from outside the production host itself?
2. Do they remain reachable from the target customer regions (`SEA`, `EU`, `RU-friendly ASN`)?
3. Can the team observe degradation historically instead of checking one-off from a laptop?

The production host cannot truthfully simulate all customer geographies, and deploy verification should not depend on persistent external-region latency. We need a separate monitoring layer that:

- uses the same real public product endpoints generated by production code paths
- executes probes from external locations
- persists the outcomes for trend analysis and incident review
- alerts operators without blocking every deploy on third-party probe instability

## Decision

Introduce a dedicated geo-probe monitoring flow based on public endpoint checks from external nodes.

### Probe source

Use the Check-Host public API as the default multi-region probe provider. The runner selects one node per required region:

- `SEA`
- `EU`
- `RU-friendly`

Selection is automatic by country priority when the provider node catalog is available, with optional explicit overrides via env:

- `GEO_PROBE_PREFERRED_NODES_SEA`
- `GEO_PROBE_PREFERRED_NODES_EU`
- `GEO_PROBE_PREFERRED_NODES_RU_FRIENDLY`

### Targets

The runner does not invent synthetic endpoints. It resolves the same public targets that real users receive:

- `MTProto` public host and port from the live service
- `WhatsApp` public host and allocated fixture port
- `VPN Amsterdam` subscription URL
- `VPN Saint Petersburg` subscription URL

Smoke fixtures are provisioned first so the geo-probe job always operates on deterministic subscription instances.

### Persistence

Persist runs and per-region observations in the database:

- `proxycraft_geo_probe_runs`
- `proxycraft_geo_probe_results`

This gives an audit trail for:

- when a probe ran
- which endpoints were checked
- from which region/node
- whether it passed, warned, failed, or was skipped
- the observed latency / HTTP status

### Scheduling and alerting

Run geo probes via a dedicated GitHub Actions workflow on a schedule and via manual dispatch. The workflow executes inside the production `proxycraft-bot` container over SSH, so it uses the same code, env, and fixture registry as the live stack.

Geo probes are **not** part of deploy-blocking verification. They are an operational monitoring signal. Failures and warnings notify `BOT_ADMINS`, but external probe flakiness must not automatically fail every deploy.

## Consequences

### Positive

- Public VPN/proxy endpoints are monitored from outside production, not just from inside the cluster.
- Operators get a persisted history of regional degradation.
- Monitoring remains aligned with real product links and fixtures instead of fake synthetic URLs.
- Deploy verification stays deterministic and focused on local production correctness.

### Trade-offs

- Geo probes depend on a third-party probe provider and its node availability.
- Regional checks become "best available" when provider nodes for a specific geography are temporarily unavailable.
- The monitoring layer is intentionally weaker than owned geo infrastructure; it is an operational baseline, not a substitute for dedicated regional probes.
